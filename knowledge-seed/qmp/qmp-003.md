# QMP-003 — Debugging Production Issues

## Q — Question
How do I systematically debug a problem in a live production system without making it worse?

## M — Model
**Observe → Hypothesize → Isolate → Fix → Verify**

The most common debugging failure mode is jumping straight to fixes without understanding
the system state. Every premature fix risks masking the real root cause or introducing a
second failure.

Production is read-only until the root cause is confirmed. Logs and metrics are the primary
evidence source.

## P — Process

### 1. Observe — establish current state
```bash
# Check service status
sudo systemctl status myservice --no-pager

# Check recent logs (last 100 lines)
journalctl -u myservice --no-pager -n 100

# Check for OOM kills or kernel errors
journalctl -k --no-pager -n 50 | grep -i "killed\|error\|oom"

# Check disk and memory
df -h && free -h
```

### 2. Hypothesize — form a specific testable theory
Write it down: _"I think X is failing because Y, which I would expect to see as Z in the logs."_
Do not skip this — it prevents thrashing.

### 3. Isolate — narrow to the specific failure
- Is the failure continuous or intermittent?
- Does it affect all users or a subset?
- Did anything deploy recently? (`git log --oneline -10`)
- Is there a pattern in the timestamps?

### 4. Fix — apply the minimal change
- Make one change at a time
- Prefer config/restart over code changes when possible
- If deploying code: commit first, then deploy via standard workflow (see QMP-001)

### 5. Verify — confirm the fix held
```bash
# Watch logs in real-time for 2–3 minutes after fix
journalctl -u myservice -f --no-pager

# Confirm the triggering error no longer appears
journalctl -u myservice --no-pager -n 50 | grep -i "error\|exception\|fail"
```

### Cap repair loops
After 2 full cycles of hypothesize→fix without resolution, stop and escalate with:
- What you tried (both attempts)
- Exact error messages
- Current system state

## Applied To
- Any server-side service, bot, or background process

## Pitfalls
- Changing multiple things at once — makes it impossible to know what fixed it
- Restarting as the first response — hides the error state needed to diagnose
- Not checking logs before and after a fix — can't verify improvement
- Fixing symptoms rather than root cause — issue recurs
