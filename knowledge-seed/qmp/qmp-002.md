# QMP-002 — Database Migration Safety

## Q — Question
How do I run a database schema migration in production without losing data or causing an outage?

## M — Model
**Migrate-then-deploy model**: run the migration first, verify it succeeded, then deploy the
application code that depends on the new schema. Never run both simultaneously.

Migrations are one-way doors. The recovery path is always a new compensating migration,
never a destructive rollback.

## P — Process

### Before migrating
1. **Back up the database** — full snapshot or pg_dump before touching schema
2. **Test migration on a copy** of production data (staging or local restore)
3. **Verify the migration is backwards-compatible** — can the current app code run against the new schema?

### Migration sequence
```bash
# Prisma (production-safe)
npx prisma migrate deploy

# SQLAlchemy / Alembic
alembic upgrade head

# Django
python manage.py migrate --run-syncdb
```

### NEVER use
```bash
# These destroy data:
npx prisma db push --accept-data-loss   # BLOCKED — wipes production data
npx prisma db push                      # May drop columns silently
```

### Post-migration verification
1. Check row counts on affected tables match expectations
2. Spot-check key records to verify data integrity
3. Run the application's healthcheck endpoint

### If migration fails mid-run
1. Do NOT retry blindly — check the migration state table first
2. Run `migrate status` (or equivalent) to see what was applied
3. Write a compensating migration to undo partial changes
4. Never manually edit the migration history table

## Applied To
- Any project using Prisma, Alembic, Django ORM, or raw SQL migrations

## Pitfalls
- Using `db push` instead of `migrate deploy` in production (Prisma) — destroys data
- Running app deploy before migration completes — new code fails on old schema
- Not backing up before migrating — no recovery path on failure
- Assuming `migrate rollback` is safe — often drops columns and data
