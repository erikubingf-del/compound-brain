# QMP-001 — Safe Production Deployment

## Q — Question
How do I deploy code changes to a live server without causing downtime or data loss?

## M — Model
**Commit-then-pull model**: changes are committed and pushed to the remote repo first,
then the server pulls and restarts. This ensures the server always runs code that exists
in version control — enabling instant rollback via `git reset --hard`.

Key principle: the server is a consumer of the repo, not a source of truth. The repo is
the source of truth.

## P — Process

### Pre-deploy checklist
1. All changes committed and pushed to the target branch
2. Tests pass locally (or CI passes)
3. No uncommitted edits in files that will be deployed

### Deploy sequence
```bash
# 1. Pull latest and restart service
ssh user@server "cd ~/project && git pull --ff-only origin main && sudo systemctl restart myservice"

# 2. Verify service is up
ssh user@server "sudo systemctl status myservice --no-pager"

# 3. Spot-check logs for startup errors
ssh user@server "journalctl -u myservice --no-pager -n 20"
```

### Rollback
```bash
# If something breaks, revert to the previous commit
ssh user@server "cd ~/project && git reset --hard HEAD~1 && sudo systemctl restart myservice"
```

### If deploy is blocked by dirty server state
```bash
# Stash or discard local server changes (only if they are not important)
ssh user@server "cd ~/project && git stash && git pull --ff-only origin main"
```

## Applied To
- Any project deployed via systemd + git pull workflow

## Pitfalls
- Deploying before pushing to remote → server pulls stale code or nothing changes
- Restarting only one service when multiple services share the updated code
- Not checking logs after restart → silent startup failures go unnoticed
- Using `git pull` without `--ff-only` → can create unwanted merge commits on server
