# QMP-004 — Feature Evaluation and Shipping

## Q — Question
How do I evaluate whether a new feature is ready to ship, and ship it safely?

## M — Model
**Gate-then-ship model**: every feature must clear a defined set of gates before reaching
production. Gates exist at three layers: correctness (does it work?), safety (can it break
things?), and reversibility (can we undo it?).

Ship small, ship often, monitor after each ship. A feature that ships in one PR is safer than
one that ships after months of accumulation.

## P — Process

### Define done before building
Write the acceptance criteria first:
- What observable behavior proves it works?
- What does failure look like?
- How will it be monitored in production?

### Pre-ship gates
| Gate | Check |
|------|-------|
| Tests pass | `pytest / npm test` exits 0 |
| No regressions | Baseline test suite still passes |
| Code reviewed | At least one other set of eyes (or a deliberate solo decision) |
| Rollback plan | What's the revert command if something breaks? |
| Monitoring in place | How will you know if it fails silently in prod? |

### Ship sequence
1. Merge to main (or deploy branch)
2. Deploy via standard workflow (see QMP-001)
3. Monitor for 5–15 minutes after deploy
4. Check error rates, key metrics, and logs

### If something breaks after ship
1. Revert immediately (`git revert HEAD` or rollback deploy)
2. Investigate in non-production
3. Do not attempt to hot-patch a broken production deploy

### Feature flags for risky changes
For high-risk features, gate behind a flag:
- Ship the code with flag=OFF
- Enable for internal users first
- Enable broadly only after internal validation

## Applied To
- Any software project shipping to production

## Pitfalls
- Defining done as "code written" rather than "behavior verified"
- Shipping without a rollback plan — makes every deploy a one-way door
- Not monitoring after deploy — silent regressions go undetected for hours
- Accumulating multiple features in one deploy — impossible to attribute failures
